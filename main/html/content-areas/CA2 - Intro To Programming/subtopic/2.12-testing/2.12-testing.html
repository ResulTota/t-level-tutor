2.12.1 - Testing Components

2.12.1.1 - Understand the reasons for testing individual components of a solution before putting them together in the final solution

Software

Verify that individual code modules and programs function correctly, handle inputs/outputs as intended, and meet their specifications.

Hardware

Ensure physical devices (e.g., sensors, storage, network adapters) operate as expected and interact reliably with software and other components.

Data

Check that data is accurate, complete, in the correct format and range, and that it flows correctly between components.

Interfaces

Confirm that components communicate correctly (software↔software, software↔hardware, user↔system) with compatible protocols and data formats.

Resulting Service (Final Product)

Validate that the integrated solution delivers the intended user experience, reliability, and performance before release.

2.12.2 - Testing Methods

Know a definition of testing methods and understand their purpose, benefits and drawbacks, and when they are used

Concept Testing

Early testing of an idea, mock-up, or prototype to gather feedback before full development.

Benefits:
  • Finds major flaws early when changes are cheap.
  • Captures stakeholder/user expectations sooner.
  • Guides priority and scope decisions.
Drawbacks:
  • Feedback can be subjective or incomplete.
  • May overlook technical feasibility issues.
  • Risk of false confidence from rough prototypes.

Unit Testing

Testing individual functions/classes in isolation.

Benefits:
  • Detects defects early and locally.
  • Simplifies debugging and refactoring.
  • Improves code quality and documentation.
Drawbacks:
  • Doesn’t verify interactions between modules.
  • Initial setup and maintenance effort.
  • Can encourage over-mocking if misused.

Boundary Testing

Focus on values at the edges of valid/invalid input ranges.

Benefits:
  • Finds common edge-case defects efficiently.
  • Strengthens input validation.
  • Small test sets with high fault yield.
Drawbacks:
  • Doesn’t cover all typical inputs.
  • May miss multi-parameter interactions.
  • Still needs complementary testing.

Integration Testing

Verify interactions between modules/services.

Benefits:
  • Catches interface and contract mismatches.
  • Builds confidence in end-to-end flows.
  • Reveals configuration/environment issues.
Drawbacks:
  • Harder to isolate root causes.
  • Complex test environments required.
  • Slower and more costly than unit tests.

Performance Testing

Assess responsiveness, throughput, and resource usage under expected load.

Benefits:
  • Identifies bottlenecks before release.
  • Helps capacity planning and tuning.
  • Improves user experience and SLAs.
Drawbacks:
  • Requires specialised tools and skills.
  • Realistic data/workloads can be hard to model.
  • Infrastructure costs for full-scale testing.

Be able to use testing methods

Students should practice applying different testing methods:

  • Use unit tests for small pieces of code.
  • Apply boundary tests to input forms.
  • Run integration tests after combining modules.
  • Conduct system tests before final delivery.

2.12.3 - Automation

Purpose of Automation Methods

Automation reduces human error and speeds up repetitive testing.

  • Macros: Automate repetitive actions in applications (e.g., Excel).
  • Scripts: Custom small programs to run automated checks.
  • Functional Testing Tools: Tools like Selenium or JUnit that run repeatable automated tests.

✅ Used when: systems are large, repetitive, or require frequent retesting (e.g., regression testing).

2.12.4 - Test Data and Test Plan

2.12.4.1 - Types of Test Data

  • Valid Data: Correct format and expected range (e.g., age = 25).
  • Invalid Data: Wrong type/format (e.g., age = “hello”).
  • Boundary Data: Values at the edge of valid ranges (e.g., age = 0, 120).
  • Erroneous Data: Clearly incorrect/extreme (e.g., age = -999).

2.12.4.2 - Creating Test Data

  • Prepare sets of valid, invalid, boundary, and erroneous inputs.
  • Use them consistently to test different components.

2.12.4.3 - Test Plan

A test plan outlines what tests will be carried out, how, and why.

Steps/Structure:

  1. Identify tests → What functions/features will be tested.
  2. Describe purpose → Why each test is needed.
  3. Identify test data → Inputs to use (valid, invalid, boundary, erroneous).
  4. Expected results → What should happen if the system works correctly.
  5. Record actual results → Compare outcomes to expected results.
